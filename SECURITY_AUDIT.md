## 风险等级说明

| 等级 | 说明 | 处理建议 |
|------|------|----------|
| 🔴 **高危** | 可能导致数据泄露、权限被接管 | 立即修复 |
| 🟡 **中危** | 一定条件下可利用 | 尽快修复 |
| 🟢 **低危** | 风险极低或需多重条件 | 选择性修复 |
| ⚪ **信息** | 非漏洞，但值得关注 | 知晓即可 |

## 一、SQL 注入风险

### ✅ 整体评价：安全

全部数据库查询使用 `PDO::prepare()` + 参数绑定，未发现字符串拼接SQL注入漏洞。

**检查要点：**
- 所有 `SELECT` / `INSERT` / `UPDATE` / `DELETE` 均使用预编译语句 ✅
- 用户输入均通过参数绑定传入（`?` 占位符）✅
- `query()` 仅用于无参数查询（`SHOW TABLES`, `SELECT setting_key` 等）✅

### 🟢 低危：Document::updateFields() 字段名拼接

**位置**: `app/Models/Document.php` 第75-83行
```php
$fields[] = "$key = ?";
$sql = "UPDATE documents SET " . implode(', ', $fields) . " WHERE id = ? AND user_id = ?";
```
`$data` 数组的**键名**直接拼接到 SQL 中。虽然 `$data` 由控制器 `AdminController.php` 构建（非直接 `$_POST`），但如果控制器扩展或重构时不小心传入用户控制的键名，可能导致注入。

**建议**: 添加字段白名单过滤。

---

## 二、XSS（跨站脚本）风险

### ✅ 整体评价：安全

模板文件统一使用 `htmlspecialchars()` 输出用户数据。

**检查要点：**
- 模板中所有 `<?= variable ?>` 均调用 `htmlspecialchars()` ✅
- 分享页面 `show.php` 中 `$htmlContent` 直接输出——但这是用户编辑的文档内容（预期行为 Markdown→HTML 渲染）✅
- 管理后台显示的日志内容已用 `htmlspecialchars()` 编码 ✅

## 三、CSRF（跨站请求伪造）

### 🟡 中危：无 CSRF Token 防护 ⚠️

**问题**: 所有管理后台的 POST 操作（创建文档、删除、修改设置等）**均无 CSRF Token 验证**。

**影响**: 如果管理员已登录后台，攻击者可以诱骗管理员访问恶意页面，从而在不知情的情况下执行删除文档、修改设置等操作。

**建议**: 
- 在后台表单中注入 `csrf_token`（`session` 中存储随机 Token）
- POST 请求必须验证 Token 匹配
- 从下次大版本（v2.0.0）开始实现

---

## 四、会话安全

### 🟡 中危：无会话超时机制

**问题**: 登录后 SESSION 永不过期（无空闲超时设置，无过期时间）

**影响**: 管理员离开电脑后，会话不会自动失效，增加会话劫持风险。

### ✅ 设备指纹验证

**位置**: `Security.php` 第36行 `validateFingerprint()`
- 记录用户登录时的 IP 段 + User-Agent 哈希
- 会话不匹配时自动销毁重建 ✅

### 🟢 建议
- 添加空闲超时（建议 30 分钟）
- 设置 `session.cookie_lifetime` 或使用 `session_set_cookie_params()` 过期时间

## 五、文件上传安全

### ✅ 整体评价：安全

**检查要点：**
1. 存储文件名使用 `Security::generateToken(16) + _ + timestamp + .ext` —— 不可预测 ✅
2. 使用 `move_uploaded_file()` —— 仅处理已上传文件 ✅
3. MIME 类型基于扩展名映射，不信任客户端提供的 `Content-Type` ✅
4. 文件存储在 `storage/files/` 目录（不在 Web 根目录下，无法直接访问）✅

### 🟢 低危：扩展名直接使用用户文件名

**位置**: `File.php` 第18行
```php
$ext = pathinfo($uploadedFile['name'], PATHINFO_EXTENSION);
```
直接从用户上传文件名获取扩展名。虽然文件名本身不可控（系统生成 token），但如果 `pathinfo()` 解析异常（如双扩展名 `file.php.jpg`），不会造成安全风险（存储路径不可预测）。

## 六、密码安全

### ✅ 整体评价：安全

**检查要点：**
- 密码使用 `password_hash(PASSWORD_DEFAULT)` → bcrypt ✅
- 验证使用 `password_verify()` → 安全比较 ✅
- 数据库存储哈希值，不存明文 ✅

### 🟢 低危：无密码复杂度策略

注册/安装时没有密码最小长度或复杂度要求。建议添加 ≥8 字符规则。

## 七、安装程序安全

### ✅ 整体评价：安全

**检查要点：**
- 安装完成后生成 `install.lock`，阻止重复安装 ✅
- `install.sql` 为固定内容，无用户输入 ✅

### 🟢 低危

## 十、性能评估

### 服务器配置

| 项目 | 值 |
|------|-----|
| CPU | 2 vCPU (Intel Xeon Platinum 8255C @ 2.50GHz) |
| 内存 | 3.6 GB |
| 磁盘 | 59 GB SSD |
| Nginx | 1.28.1 (worker_processes auto → 2 workers) |
| PHP | 8.5.2 (PHP-FPM, 50 max children) |
| MySQL | 同机（127.0.0.1:3306） |

### 分享链接吞吐估算

| 场景 | 估算值 | 说明 |
|------|--------|------|
| **同时在线** | ~400-500 人 | Nginx 2 workers × 51200 connections，PHP-FPM 50 workers |
| **每分钟访问** | ~3,000-6,000 次 | 平均响应 200ms，50 workers × 60s / 0.2s |
| **每小时访问** | ~180,000-360,000 次 | 按分钟估算放大 |
| **每日访问** | ~430万-860万次 | 理论峰值，真实场景受网络带宽影响 |

**瓶颈分析：**
1. **PHP-FPM max_children=50**——同时只能处理50个请求，队列中的请求等待。如需更高并发，可调至 100-200（取决于每个请求内存占用 ~20-30MB）
2. **MySQL 同机**——分享访问每次需查询 DB，MySQL 连接数上限需同步调整
3. **磁盘 I/O**——文件分享需要读磁盘，SSD 平均 10ms/次

**优化建议：**
- 分享页面启用 Nginx 缓存（静态化）
- PHP-FPM max_children 调至 80-100
- MySQL 添加 `max_connections = 200`

## 风险评估汇总

| # | 风险项 | 等级 | 修复优先级 |
|---|--------|------|-----------|
| 1 | 无 CSRF Token | 🟡 中危 | 下版本 |
| 2 | 无会话超时 | 🟡 中危 | 下版本 |
| 3 | 字段名拼接注入 | 🟢 低危 | 建议修复 |
| 4 | 无密码复杂度策略 | 🟢 低危 | 建议修复 |
| 5 | Session 未设置 cookie 参数 | 🟢 低危 | 建议修复 |

### 影响评估

当前 MemoBox 主要用于中等规模团队的知识管理场景（非公开注册系统），实际攻击面有限：
- 后台需管理员主动登录，无法被外部直接访问
- 分享链接仅限 Token 持有者访问
- 所有上传文件路径不可预测
- 安装完成后无法重复安装

基于以上场景，**当前安全风险可控，无需紧急修复**。
